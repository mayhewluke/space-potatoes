Movement: W/A/D

Space Potatoes? Home-cooked wha?

This game is a (shoddy) remake of the 1979 Atari game "Asteroids". Folks like to say that asteroids look like potatoes (in space). And a game needs a name, so since I'm so witty, "Space Potatoes" was born.

So what's the deal with "Home-cooked FRP"? FRP stands for "Functional Reactive Programming". I'm not going to get into the minutia of that here, because that's complicated, but it's basically like taking Functional Programming (which is itself a fantastic way to program) and adding on the ability to react to events (eg. user input) in an equally elegant way. In other words, it's the best damn way to program I've ever found.

I've used various Reactive and Functional Reactive systems in the past, and have normally been satisfied with just knowing how to use them, even though how the systems worked was still black magic to me. Recently, though, I came across a couple YouTube videos that inspired me. One was making a remake of Asteroids in JavaScript. I'd never made a game before, or worked with the canvas at all, and it seems like a fun way to learn something new - and being a game, something more exciting to build than just another web app. The other was a series on building a Reactive system from scratch. It made the basic building blocks made sense, and inspired me to not just try Reactive Programming again, but actually try implementing it. To really dig in, get grubby, and figure out what's going on under the hood.

And thus Space Potatoes was born. The game is obviously very simplistic and lacking, because for me the real game was actually building the underlying FRP system.

Feel free to check out the source for said system on my Github. Being a free-time exploration for the sake of learning and becoming stronger professionally, it's obviously not 100% feature-complete or production-ready, but it is usable and clean enough to be readable.

One caveat: TypeScript doesn't have great support for type inference with functional programming styles, so there are a number of cases where the type system (and its shortcomings) make the syntax more unwieldy than it should be. Ultimately there's a tradeoff between JS/TS and first-class functional programming languages, and as a web developer, JS/TS is what I get paid to write in, so it's what made sense to use for this exploration. Were we wanting proper, first-class support for this style of programming, something like Haskell would probably be a better bet in the long-term.
